# Tetris Implementation Plan with `superlighttui`

## 1. 목표

Rust와 `superlighttui = 0.15.7`를 사용해 터미널에서 동작하는 싱글 플레이 테트리스를 구현한다.

핵심 목표:

- 즉시 실행 가능한 MVP를 먼저 완성한다.
- `superlighttui`의 immediate-mode 방식에 맞게 상태와 렌더링을 단순하게 유지한다.
- 게임 로직은 UI와 분리해서 테스트 가능하게 만든다.

## 2. 전제와 방향

`superlighttui`는 `slt::run(|ui| { ... })` 형태로 프레임마다 UI 클로저를 다시 실행하는 구조다. 따라서 테트리스는 다음 원칙으로 구현한다.

- 게임 상태는 클로저 바깥의 로컬 상태로 유지한다.
- 입력 처리와 시간 누적은 매 프레임 클로저 안에서 갱신한다.
- 레이아웃은 `row()` / `col()` 기반으로 만든다.
- 플레이필드는 braille `canvas` 대신 고정 셀 그리드를 사용한다.
  이유: 테트리스는 정확한 셀 단위 충돌과 색상 블록 표현이 중요해서 일반 텍스트 셀 렌더링이 더 단순하고 안정적이다.

## 3. MVP 범위

MVP에 포함:

- 10x20 보드
- 7종 테트로미노
- 좌우 이동
- 소프트 드롭
- 하드 드롭
- 회전
- 바닥/블록 충돌 판정
- 라인 제거
- 점수, 라인 수, 레벨
- 게임 오버
- 재시작
- 일시정지
- 다음 블록 표시

MVP에서 제외:

- 홀드
- SRS wall kick 완전 구현
- 고스트 피스
- 애니메이션 연출
- 저장/로드
- 사운드

제외 항목은 MVP 이후 확장 단계에서 추가한다.

## 4. 추천 구조

초기에는 파일 수를 과도하게 늘리지 말고, 아래 정도로 분리한다.

```text
src/
  main.rs
  app.rs
  game/
    mod.rs
    board.rs
    piece.rs
    rules.rs
```

역할:

- `main.rs`
  `superlighttui` 실행 진입점
- `app.rs`
  앱 상태, 화면 상태, 입력 매핑, 프레임 타이밍 관리
- `game/board.rs`
  보드 데이터, 충돌 판정, 고정, 라인 삭제
- `game/piece.rs`
  테트로미노 종류, 회전 상태, 좌표 계산
- `game/rules.rs`
  점수, 레벨, 낙하 속도 계산 규칙

작게 시작하고 싶다면 1차 구현은 `main.rs + game.rs` 두 파일만으로 시작해도 된다. 다만 테스트를 붙이려면 위 구조가 더 낫다.

## 5. 상태 모델

### 앱 상태

```rust
enum ScreenState {
    Title,
    Playing,
    Paused,
    GameOver,
}
```

```rust
struct App {
    screen: ScreenState,
    game: GameState,
    last_frame: Instant,
    gravity_accumulator: Duration,
    input_cooldown: Duration,
}
```

### 게임 상태

```rust
struct GameState {
    board: Board,
    current: ActivePiece,
    next: TetrominoKind,
    score: u32,
    lines: u32,
    level: u32,
    rng: Randomizer,
}
```

추가 메모:

- 랜덤 생성은 최소 구현에서는 단순 RNG로 시작한다.
- 이후 공정성을 위해 7-bag 방식으로 바꾸는 것이 좋다.

## 6. 핵심 로직 설계

### 6.1 보드 표현

- 보드는 `[[Option<Cell>; 10]; 20]` 또는 `Vec<Vec<Option<Cell>>>`로 관리한다.
- Rust에서 다루기 단순한 쪽은 고정 크기 배열이다.
- 각 셀은 빈 칸 또는 블록 색상/종류를 가진다.

예시:

```rust
type Board = [[Option<TetrominoKind>; BOARD_WIDTH]; BOARD_HEIGHT];
```

### 6.2 피스 표현

- `TetrominoKind`: `I, O, T, S, Z, J, L`
- `Rotation`: `0, 1, 2, 3`
- `ActivePiece`: 현재 종류, 회전 상태, 기준 좌표

중요한 판단:

- 각 피스의 상대 좌표를 회전별로 미리 정의해둔다.
- 실시간 회전 계산보다 디버깅이 쉽고 테스트가 단순하다.

### 6.3 충돌 판정

다음 상황을 검사한다.

- 보드 좌측/우측 경계 초과
- 바닥 초과
- 이미 고정된 블록과 겹침

필수 함수:

- `can_move(dx, dy) -> bool`
- `can_rotate() -> bool`
- `lock_piece()`
- `clear_lines() -> u32`
- `spawn_next_piece() -> bool`

`spawn_next_piece()`가 실패하면 게임 오버다.

### 6.4 낙하와 타이밍

`superlighttui`가 프레임 기반으로 클로저를 다시 실행하므로, 게임 진행은 델타 타임 누적 방식으로 처리한다.

- 매 프레임 `now - last_frame` 계산
- 누적 시간이 현재 레벨의 gravity interval 이상이면 한 칸 낙하
- 더 이상 낙하 불가면 고정 후 라인 삭제와 새 피스 생성

초기 gravity 예시:

- 레벨 1: 700ms
- 레벨 증가 시 점진적으로 감소

간단한 구현 순서:

1. 기본 중력만 먼저 구현
2. 하드 드롭 추가
3. 소프트 드롭 속도 보정
4. 필요하면 lock delay 추가

MVP에서는 lock delay를 생략해도 된다.

## 7. 입력 계획

추천 키맵:

- `Left` / `h`: 왼쪽 이동
- `Right` / `l`: 오른쪽 이동
- `Down` / `j`: 소프트 드롭
- `Up` / `k` 또는 `x`: 회전
- `Space`: 하드 드롭
- `p`: 일시정지
- `r`: 재시작
- `q`: 종료

입력 처리 원칙:

- 이동/회전은 1프레임 1회 처리
- 소프트 드롭은 누르고 있는 동안 중력보다 빠르게 낙하
- 하드 드롭은 즉시 바닥까지 이동 후 고정

## 8. 화면 구성

기본 레이아웃:

```text
+--------------------------------------------------+
| TETRIS                                           |
|                                                  |
| +--------------------+  +----------------------+ |
| |                    |  | Score                | |
| |     Playfield      |  | Level                | |
| |      10 x 20       |  | Lines                | |
| |                    |  | Next Piece           | |
| +--------------------+  | Controls             | |
|                         +----------------------+ |
+--------------------------------------------------+
```

구성 계획:

- 바깥 컨테이너: 타이틀 + 본문
- 본문 `row()`
  - 왼쪽: 플레이필드
  - 오른쪽: 점수 패널, 다음 블록, 조작 안내

렌더링 방식:

- 한 셀은 너비 2칸 문자열 또는 배경색 블록으로 표현
- 빈 칸은 어두운 배경 또는 점 문자
- 고정 블록과 현재 피스를 합성해서 그린다

권장 표현:

- 셀 하나를 `"  "` + 배경색으로 그리기
- 테두리는 `bordered()` 사용

## 9. 개발 단계

### 단계 1. 프로젝트 골격

- `main.rs`에서 `slt::run` 연결
- `App` 상태 생성
- 타이틀/빈 보드 UI 출력

완료 기준:

- 프로그램 실행 시 빈 테트리스 화면이 보인다.

### 단계 2. 보드와 피스

- 보드 자료구조 구현
- 현재 피스 스폰
- 플레이필드에 피스 표시

완료 기준:

- 시작 시 피스 1개가 정상 위치에 렌더링된다.

### 단계 3. 입력과 이동

- 좌우 이동
- 회전
- 소프트 드롭
- 하드 드롭

완료 기준:

- 벽과 블록 충돌을 지키면서 피스를 조작할 수 있다.

### 단계 4. 게임 진행

- 자동 낙하
- 바닥 고정
- 다음 피스 스폰
- 게임 오버

완료 기준:

- 입력 없이도 게임이 진행되고, 보드가 차면 게임 오버가 된다.

### 단계 5. 점수와 라인 제거

- 가득 찬 줄 탐지
- 라인 삭제 후 위 블록 낙하
- 점수/레벨 반영

완료 기준:

- 라인을 지우면 보드와 점수가 올바르게 갱신된다.

### 단계 6. polish

- 일시정지
- 재시작
- 다음 블록 패널
- 타이틀/게임오버 메시지

완료 기준:

- 처음부터 끝까지 한 판 플레이 가능한 UX가 완성된다.

## 10. 테스트 계획

UI보다 게임 로직 테스트가 우선이다.

필수 단위 테스트:

- 피스 회전 좌표가 예상과 일치하는지
- 벽 충돌이 정상 차단되는지
- 블록 충돌이 정상 차단되는지
- 라인 1개/여러 개 삭제가 정확한지
- 새 피스 스폰 실패 시 게임 오버가 되는지

수동 확인 항목:

- 빠른 방향키 입력 시 오작동 없는지
- 터미널 크기가 작을 때 레이아웃이 무너지지 않는지
- 일시정지와 재시작이 정상 동작하는지

## 11. 리스크와 대응

### 입력 반복 처리

터미널 키 반복 속도에 따라 조작감이 달라질 수 있다.

대응:

- MVP는 단순 처리
- 필요하면 입력 쿨다운 또는 DAS/ARR 개념 추가

### 회전 규칙 단순화

완전한 SRS가 없으면 벽 근처 회전이 어색할 수 있다.

대응:

- MVP는 단순 회전
- 이후 wall kick 테이블 추가

### 렌더링 가독성

터미널 폰트에 따라 셀 비율이 달라 보일 수 있다.

대응:

- 셀 너비 2칸 유지
- 테두리와 배경색 대비를 강하게 준다

## 12. 확장 로드맵

MVP 이후 우선순위:

1. 7-bag 랜덤
2. 고스트 피스
3. 홀드 기능
4. SRS wall kick
5. 시작 화면과 난이도 선택
6. 최고 점수 저장

## 13. 완료 정의

아래를 만족하면 1차 완료로 본다.

- 실행 즉시 플레이 가능하다.
- 한 판을 끝까지 진행할 수 있다.
- 점수, 레벨, 라인 수가 보인다.
- 게임 오버 후 재시작 가능하다.
- 핵심 게임 로직 테스트가 존재한다.

## 14. 바로 다음 작업 순서

권장 시작 순서:

1. `App`, `GameState`, `Board`, `TetrominoKind` 타입 정의
2. 빈 보드와 단일 피스 렌더링
3. 이동/회전/낙하 구현
