# Tetris Study Notes - 2026-03-25

## 핵심 정리

- 현재 `Tetris` 구현은 `frame -> render` 구조다.
- `frame`은 상태를 바꾼다.
  - 입력 처리
  - 중력 진행
  - 충돌 검사
  - 락 처리
  - 줄 삭제 처리
- `render`는 현재 상태를 화면에 그린다.

## 루프 구조

- `App` 안에 직접 `loop`가 있는 것은 아니다.
- 반복문은 `superlighttui`의 `run_with(...)` 내부에 있다.
- `slt`가 매 프레임 클로저를 다시 호출하고, 그 안에서 `app.frame(ui)`가 실행된다.

흐름:

```text
slt run loop
  -> app.frame(ui)
    -> game.frame(ui, delta)
      -> update
      -> render
```

## Tick에 대한 이해

- 현재 `tick`이라는 이름의 함수는 없다.
- 대신 `frame` 내부에서 `delta`와 gravity accumulator로 같은 역할을 한다.
- 즉 `tick`이 없는 것이 아니라 `frame` 안에 흡수된 구조다.

## Tetris 렌더 방식

- 현재 `Tetris`는 `draw`로 절대좌표에 블록을 직접 찍는 방식이 아니다.
- 보드 전체를 셀 단위로 다시 렌더한다.
- 각 `(x, y)` 셀마다 다음 중 하나를 판단한다.
  - 현재 블록
  - 고정된 블록
  - 고스트 블록
  - 빈칸
- 그 결과를 `"██"`, `"░░"`, `"· "` 같은 텍스트로 출력한다.

즉:

```text
특정 블록을 직접 이동해서 그림
```

보다

```text
보드 전체를 다시 만들고
해당 셀만 블록처럼 보이게 렌더
```

가 현재 구조에 더 가깝다.

## `ml`, `mt`에 대한 이해

- `ml` = `margin-left`
- `mt` = `margin-top`
- 이것은 게임 좌표가 아니라 레이아웃 여백이다.
- 그래서 테트리스 같은 격자 게임에서 좌표 표현용으로 쓰기에는 부정확하다.

정리:

- 간단한 UI 배치에는 `ml`, `mt`
- 보드 좌표 표현에는 셀 기반 렌더 또는 `draw`

## `draw` 사용 시 주의

- `draw`에 넘기는 클로저는 `'static`이어야 한다.
- 그래서 클로저 안에서 `self`를 직접 참조하면 borrow 에러가 날 수 있다.
- 필요한 값은 클로저 밖에서 꺼내서 값으로 복사한 뒤 `move`로 넘기는 편이 안전하다.

## `MyTetris`에 대한 방향

- 학습용 구현은 먼저 작은 상태를 명확히 두는 것이 좋다.
  - 보드
  - 현재 블록 위치
  - `frame`
  - `render`
- 처음부터 복잡한 렌더보다, 보드 전체를 다시 그리는 방식으로 시작하는 것이 테트리스 구조를 이해하기 쉽다.
